Architecture Refactor

Context (arnetheduck, 2026-03-26): The vnext/declarative parsing code was layered on top of the old architecture instead of integrated. The codebase is a path-dependent outcome of adding vnext on top of legacy. If you wrote nimble from scratch today with declarative-first intent, the code would not look like this. The goal is to reframe the architecture so declarative parsing is the default and VM parsing is the fallback — not the other way around. There should be no "vnext" module by the end — its logic should be assimilated into the natural modules.

Motivation

The core architectural problem: getPkgInfo (packageparser.nim) was the canonical entry point for package metadata. Then the declarative parser was added, but only wired through nimblesat.nim. Now there are two parallel parsing paths:

  • getPkgInfo() / getPkgInfoFromFile() — called from 12+ files, uses the VM parser
  • getPkgInfoFromDirWithDeclarativeParser() — called from vnext.nim, nimblesat.nim, download.nim, developfile.nim

These should be one entry point with a fallback chain: declarative → VM. All callers should go through the same function. This will eliminate an entire class of bugs where one code path sees different package info than another.

How Nim Resolution Works Now

Nim is resolved in two passes, which is the correct design — you need some nim before you can parse nimble files to feed the SAT solver.

# Pass 1: Bootstrap (before SAT solving)
# Called once in main. Finds a nim to use for parsing nimble files.
getBootstrapNimResolved(options) → nimBin: string
  → getNimFromSystem()     # 1. System nim (PATH / --nim: flag)
  → getNimBinariesPackages() # 2. Previously downloaded nims
  → getOfficialReleases()   # 3. Download latest release

# Pass 2: SAT resolution (vnext actions only)
# May refine nim based on lock file or project requirements.
resolveAndConfigureNim(rootPkg, pkgList, options, nimBin)
  → resolveNim()  # SAT-based nim selection
  → sets options.nimBin # for the few remaining consumers

# Non-vnext actions (publish, tasks, check, dump, init)
# Only the bootstrap result is needed — no SAT solving.

The nimBin: string from bootstrap is passed as a parameter through the entire pipeline. options.nimBin is still written in solvePkgs as a bridge for useNimFromDir (1 caller: install when installing nim itself).

Phase 2: Unified Package Parsing Done

Create a single canonical getPkgInfo that tries parsers in order: declarative → VM. All callers converge here. nimBin is now passed as a parameter (Phase 1), so the unified function receives it directly.

Target: Single Entry Point

proc getPkgInfo(dir: string, options: Options,
               kind: PackageInfoKind = pikRequires): PackageInfo =
  ## Canonical entry point. Tries declarative parser first,
  ## falls back to VM parser if declarative fails or if pikFull requested.
  let nimbleFile = findNimbleFile(dir)

  # Step 1: Always try declarative first
  try:
    result = parseDeclarative(nimbleFile, options)  # pikRequires
  except:
    result = parseVm(nimbleFile, options)            # pikFull (fallback)
    return

  # Step 2: If caller needs full info and declarative succeeded,
  # upgrade via VM for fields only NimScript can provide (bin, tasks, hooks)
  if kind == pikFull:
    result = upgradeToFull(result, nimbleFile, options)

Steps

  1. Create the unified getPkgInfo in packageparser.nim (or a new packageinfo.nim — the parsing entry point)
  2. Migrate declarativeparser.nim to export a simple parseDeclarative(nimbleFile, options): PackageInfo
  3. Keep the existing VM parser as parseVm(nimbleFile, options): PackageInfo
  4. Replace all direct calls to getPkgInfoFromDirWithDeclarativeParser and getPkgInfoFromFile with the unified entry point
  5. Audit every toFullInfo call — replace with getPkgInfo(dir, options, pikFull) where needed
Key insight: In a declarative-first world, most callers only need pikRequires (name, version, dependencies). Only build/install/task-running needs pikFull. The declarative parser handles 95% of use cases without spinning up a NimScript VM.

Phase 3: Decouple Parser from SAT Done

Currently nimblesat.nim imports and directly calls the declarative parser for version discovery. Parsing packages is not a SAT concern — the SAT solver should receive a pre-built version table, not build it by parsing packages itself.

Current Flow (Wrong)

nimblesat.collectAllVersions()
  → nimblesat.processRequirements()
    → nimblesat.downloadMinimalPackage()
      → declarativeparser.getPkgInfoFromDirWithDeclarativeParser()
      → packageparser.getPkgInfoFromFile()  (fallback)
  → builds pkgVersionTable internally
  → passes to SAT formula builder

Target Flow (Clean)

# 1. Build version table (uses unified getPkgInfo, nothing SAT-specific)
versionTable = buildVersionTable(rootPkg, options)
  → for each requirement, discover versions via getPkgInfo()
  → returns PkgVersionTable

# 2. SAT solver receives ready-made data
solution = solveSAT(versionTable)
  → pure constraint solving, no I/O, no parsing

Steps

  1. Extract collectAllVersions / processRequirements / downloadMinimalPackage out of nimblesat.nim into a new module (e.g. versiondiscovery.nim or fold into packageinfo.nim)
  2. These functions should call the unified getPkgInfo from Phase 2
  3. nimblesat.nim shrinks to pure SAT logic: formula building, solving, result extraction
  4. The SAT solver receives PkgVersionTable as input, returns seq[SolvedPackage] as output

Motivation

CI timing shows git clone operations account for ~55% of total test runtime. The same packages are cloned repeatedly across tests because each test clears installDir and re-downloads from scratch.

The top 3 suites (issues 400s, SAT solver 391s, lock file 225s) take ~1016s combined. Most time is cloning the same packages from GitHub.

Implementation Approach

Recommended: NIMBLE_PKGCACHE env var as read-only fallback cache.

Add a NIMBLE_PKGCACHE env var that nimble checks as a read-only fallback cache. The test infrastructure sets this to a pre-populated directory.

  • No changes to test code or cleanDir logic
  • Each test still gets a clean installDir — behavioral purity preserved
  • The fallback cache is read-only, so tests can't corrupt it
  • Useful outside tests too (CI pipelines, monorepos)

The nimble change is small: in the clone/fetch logic (download.nim), before doing a fresh git clone, check if NIMBLE_PKGCACHE is set and contains a cached repo for this URL. If so, do a local git clone --reference from the cached repo (near-instant).

mkdir myproject && cd myproject
nimble init          # creates myproject.nimble

Add a dependency to myproject.nimble:

nimble develop chronos --withDependencies

Code Architecture

Without --with-dependencies

runVNext()                                   # nimble.nim
  +- setNimBin()                             # Find/configure nim
  +- develop(options, nimBin)                 # Clone requested packages only
  |   +- installDevelopPackage()              # Download/clone to vendor/
  |   |   +- developFromDir()                # Run hooks, display success
  |   +- updateDevelopFile()                  # Write nimble.develop
  +- return                                  # No solving needed

With --with-dependencies (solve-first)

runVNext()                                   # nimble.nim
  +- setNimBin()                             # Find/configure nim
  +- develop(options, nimBin)                 # Clone ONLY explicit packages (no BFS)
  |                                          # e.g. "develop PackageB" clones PackageB
  |                                          # For project-dir develop: nothing to clone
  +- solvePkgs()                             # SAT solver resolves ALL deps
  |   +- initPkgList()                       # Scans pkgs2 + any vendor/ already cloned
  |   +- resolveAndConfigureNim()
  |   +- solvePackages()                     # Uses pkgcache metadata, not vendor/
  |       +- cacheToPackageVersionTable()    # Load tagged_versions.json
  |       +- collectAllVersions()            # Download metadata to pkgcache
  |       +- getSolvedPackages()             # SAT constraint solving
  +- developFromSolution()                   # Clone solved deps into vendor/
  |   +- for each solvedPkg (skip nim, skip root):
  |   |   +- installDevelopPackage(exact ver) # Clone to vendor/
  |   +- updateDevelopFile()                  # Write nimble.develop
  +- setup()                                 # Write nimble.paths from satResult.pkgs
Solve-first design: The SAT solver resolves dependencies using metadata from pkgcache (shallow clones used only for reading .nimble files). Only after the full dependency graph is solved does developFromSolution() clone the exact resolved versions into vendor/. This avoids the old chicken-and-egg problem and fixes the hasVersion shadowing bug where tagged_versions.json cache caused the solver to pick pkgs2 versions over vendor packages.

Rebuild Considerations

  1. Chicken-and-egg with SAT solver. Fixed. The solve-first refactor resolves this: the SAT solver now runs before cloning, using pkgcache metadata. developFromSolution() then clones the exact solved versions into vendor/. This also fixed the hasVersion shadowing bug where tagged_versions.json cache caused the solver to pick pkgs2 versions instead of vendor packages. See Code Architecture above.
  2. Too many state files. Four files interact: nimble.develop, nimble.lock, .nimble.sync, and nimble.paths. Consider merging sync into lock (add a syncedRevision field).
  3. Recursive develop includes add complexity. nimble.develop supports includes creating a tree with reference counting. Rarely used. A flat list of vendor paths would be simpler.
  4. isLink is fragile. The develop vs installed distinction hinges on a boolean that must be set correctly in multiple code paths. A more robust approach: explicit package source enum (psInstalled, psDevelop, psCache).
  5. Global develop is a separate mechanism. Creates link files in ~/.nimble/links/, completely separate from local vendor/. Consider unifying.
  6. Make nimble.paths deterministic. Sort paths alphabetically. Ensure nimble setup is idempotent.

New Commands

nimble fetch Todo

Users have no way to update unconstrained deps (requires "pkg") to their latest versions. nimble install skips packages already in pkgs2/, and the tagged_versions.json cache + pkgcache clones prevent discovery of new upstream tags. nimble upgrade only regenerates the lock file — it doesn't re-fetch or reinstall.

Problem

Three layers of caching prevent updates:

  1. tagged_versions.json — cacheToPackageVersionTable() loads stale version data into the SAT solver before it even starts resolving
  2. pkgcache clones — isCacheValid() returns true immediately for verAny ranges, so stale clones are reused without fetching new tags
  3. pkgs2/ entries — hasVersion() finds existing installations and skips re-downloading

Proposed behavior

# Update all dependencies to latest versions
nimble fetch

# Update specific packages
nimble fetch packagea chronos

Implementation steps:

  1. Clear tagged_versions.json entries for target packages (or all if no args)
  2. Delete or git fetch the pkgcache clones so new tags are discovered
  3. Remove pkgs2/ entries for those packages
  4. Re-resolve and install

PubGrub-style error messages Todo

When dependency resolution fails, the current SAT solver produces minimal error messages like "Dependency resolution failed: requirement 'b >= 0.5.0' required by 'a 3.0' could not be satisfied." This tells you which constraint failed but not why — the chain of decisions that led to the conflict.

PubGrub (used by Dart's pub, Rust's cargo via pubgrub-rs) solves this by tracking incompatibilities — each resolution failure remembers the derivation chain. When resolution fails, it can produce human-readable explanations like:

Because chronos >= 4.0 depends on stew >= 0.4
  and bearssl < 0.2 depends on stew < 0.4,
  chronos >= 4.0 is incompatible with bearssl < 0.2.

And because myproject depends on both chronos >= 4.0 and bearssl 0.1.8,
  version solving failed.

Approach: hybrid SAT + PubGrub

Keep the SAT solver for resolution (it works), add PubGrub only for error reporting:

  • Happy path (resolution succeeds): SAT solver runs as today, no change
  • Error path (resolution fails): run PubGrub on the same pkgVersionTable inputs to produce a human-readable derivation of the conflict

PubGrub is purpose-built for explaining version conflicts via incompatibility tracking. Rather than hacking clause tracking into the SAT solver (which wasn't designed for it), a standalone PubGrub implementation gives correct error messages by design. The glue code (mapping nimble's package model onto PubGrub's abstraction) is the same effort either way.

Entry point: getSolvedPackages() in nimblesat.nim — when the SAT formula is unsatisfiable, call into the PubGrub module with the same version table to generate the error message.
